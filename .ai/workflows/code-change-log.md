# Code Change Log Workflow

> 코드 작성/수정이 발생한 세션에서 작성하는 표준 설명 문서 절차입니다.

## 원칙

- 코드 변경이 있었던 세션에서는 반드시 작성한다
- "무엇을"보다 "왜"를 우선해 기록한다
- 나중에 작성자가 아닌 사람이 봐도 의도를 이해할 수 있어야 한다
- 추측이 아니라 실제 결정과 근거를 기록한다

## 절차

### 1. 문서 파일 선택

- 경로: `docs/code-change-notes/YYYY-MM-DD.md`
- 해당 날짜 파일이 없으면 새로 생성한다
- 있으면 기존 파일 하단에 새로운 세션 섹션을 추가한다

### 2. 변경 배경 기록

- 문제/요구사항이 무엇이었는지
- 변경 전 코드의 한계 또는 위험
- 이번 변경의 목표

### 3. 변경 의도 기록

- 왜 이 접근을 선택했는지
- 대안은 무엇이었고 왜 제외했는지 (있을 때만)
- 트레이드오프(성능, 복잡도, 유지보수성 등)

### 4. 코드 단위 설명

- 파일/함수 단위로 변경 이유를 설명한다
- 핵심 로직은 동작 흐름을 짧게 정리한다
- 리뷰어가 "왜 이렇게 작성했는가"를 답할 수 있어야 한다

### 5. 영향과 검증 기록

- 영향 범위(기능/모듈/사용자 영향)
- 호환성/회귀 리스크
- 실행한 테스트/검증과 결과

## 작성 규칙

- 템플릿: `.ai/templates/code-change-log.md`
- 과도한 장문보다 명확한 근거 중심으로 작성한다
- 민감 정보(토큰, 비밀값, 개인정보)를 기록하지 않는다
