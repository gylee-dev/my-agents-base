# Work Log Workflow

> 사용자와의 작업을 마무리할 때 작성하는 표준 작업 일지 절차입니다.

## 원칙

- 모든 작업 세션 종료 시 작업 일지를 반드시 남긴다
- 일지는 사실 기반으로 간결하게 작성한다
- 완료/미완료를 명확히 구분한다
- 테스트/검증 결과는 실행한 것만 기록한다
- 코드 작성/수정이 있었다면 코드 변경 의도 문서와 함께 남긴다

## 절차

### 1. 일지 파일 선택

- 경로: `docs/work-journal/YYYY-MM-DD.md`
- 해당 날짜 파일이 없으면 새로 생성한다
- 있으면 기존 파일 하단에 새로운 세션 섹션을 추가한다

### 2. 세션 식별 정보 기록

- 작업 시간(시작/종료 또는 기록 시각)
- 요청 요약
- 관련 브랜치/이슈/PR (있을 때만)

### 3. 수행 내용 기록

- 실제로 수행한 변경 사항
- 수정한 파일 목록
- 설계/구현에서의 핵심 결정 사항

### 4. 검증 결과 기록

- 실행한 테스트/린트/빌드 명령
- 성공/실패 여부
- 실패 시 원인과 현재 상태

### 5. 잔여 작업 기록

- 후속 작업 TODO
- 사용자 확인이 필요한 항목
- 알려진 리스크

### 6. 코드 변경 의도 문서 연결 (코드 변경 시 필수)

- 코드 작성/수정이 있었으면 `docs/code-change-notes/YYYY-MM-DD.md`를 작성한다
- 작성 절차는 `.ai/workflows/code-change-log.md`를 따른다
- 작업 일지에 코드 변경 의도 문서 경로를 함께 남긴다

## 작성 규칙

- 템플릿: `.ai/templates/work-log.md`
- 추측이나 계획만 작성하지 않는다 (실행한 내용 중심)
- 민감 정보(토큰, 비밀값, 개인정보)를 기록하지 않는다
